---
name: ZuruSasa mobile Supabase wiring
description: How the Expo app reaches the shared Supabase backend and auth choice
---

- The Expo app talks to Supabase directly (no api-server). Supabase env vars are mapped in the artifact's package.json dev/build scripts: `EXPO_PUBLIC_SUPABASE_URL=$VITE_SUPABASE_URL EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY=$VITE_SUPABASE_PUBLISHABLE_KEY` — Metro only exposes EXPO_PUBLIC_* vars, and this avoids ever reading secret values.
- **Why:** the same Supabase project backs web and mobile; the publishable key is public by design so shell-level mapping is safe.
- Mobile auth uses email OTP (signInWithOtp + verifyOtp type 'email') because web-style OAuth redirects and phone OTP flows are web-oriented; sessions persist via AsyncStorage.
- Reels query relies on the FK alias `profiles!reels_user_id_profiles_fkey` — keep in sync with the web app's useReels if the schema changes.
- Install expo-* packages with native code via `pnpm exec expo install <pkg>` (matches `expo/bundledNativeModules.json`), never `pnpm add <pkg>@latest`. **Why:** a newer JS package calls native modules with different signatures than the SDK's Expo Go build ships — e.g. expo-video 57.x on SDK 54 crashed Android with "Received 3 arguments, but 1 was expected"; SDK 54 needs ~3.0.x.
- The Zuru Agent chat calls the deployed Supabase edge function `city-pulse-ai` (shared with web). That function wraps upstream Gemini failures as HTTP 500 (e.g. 503 "high demand" spikes, transient). **How to apply:** when chat "breaks", curl the function directly with the publishable key before blaming client wiring; an error bubble in the app is the intended graceful path.
- E2E testing gotcha: the Expo app lives on its own dev domain (`$REPLIT_EXPO_DEV_DOMAIN`), NOT the shared proxy. Give the testing agent the literal URL — a bare "path /" sends it to the ZuruSasa WEB app. Since the mobile Discover/tab parity work, the two apps are visually near-identical at phone width (same "Pulse/Discover/Log In" nav, same "Zuru Agent" chat), so the URL in the address bar is the ONLY reliable identity marker.
- If a tester (or curl) gets HTTP 502 on the Expo dev domain, the expo workflow has died (status NOT_STARTED) — restart the workflow and curl until 200 before re-dispatching; don't debug app code first. The workflow can die silently between your own screenshot and the test run.
- Theme toggle uses `Appearance.setColorScheme()`: flips the whole app on native, but is a no-op on the RN-web dev preview — "toggle does nothing" in a browser test is expected, not a bug.
- Supabase auth providers (shared web+mobile): PHONE OTP is configured and reaches the code step; the EMAIL mailer intermittently 500s with a raw "{}" body (rate limits on the built-in mailer). Map unreadable bodies to a friendly message; don't debug client wiring first. Passkey (`signInWithPasskey`) doesn't exist in supabase-js — the web button errors too; mobile mirrors that.
- Web visual parity gotcha: the web app styles ALL h1–h6 in Instrument Serif via a global CSS rule (src/index.css), and shadcn inputs keep a `bg-background` fill strip — page-level component code won't show either. Check global CSS before assuming a heading is DM Sans.
- Native Google Sign-In: `@react-native-google-signin/google-signin` FREE module = native Play-services account picker (its Credential Manager "Universal" variant is sponsor-gated). Must lazy-`require` it — a static import crashes web/Expo Go bundles at startup (TurboModule missing). idToken audience = the WEB client ID passed as `webClientId`; Supabase `signInWithIdToken` accepts it because that's the client ID already configured on the Google provider. The Android OAuth client (package + SHA-1) just has to exist; nothing from it is embedded.
- Android signing: keystore committed at credentials/android/release.keystore + credentials.json (EAS "credentialsSource: local"). SHA-1 8E:E7:E0:5A:52:6C:D2:FC:12:80:04:FE:88:A2:11:B4:65:82:38:90 is registered with Google — losing the keystore or switching to EAS-managed keys breaks Google Sign-In until the new SHA-1 is added. Play App Signing re-signs: its SHA-1 needs adding too at publish time.
- EAS cloud builds do NOT see repl env vars — EXPO_PUBLIC_* values must be in eas.json `env` blocks (Supabase URL/key are stored as Replit secrets, so the user pastes them; agent can't copy secret values).
- gitPull/gitPush callbacks RETURN error objects (`{success:false, code:"CLI_ERROR", ...}`) instead of throwing — try/catch won't catch a failed pull; check `.success` explicitly. A pull may report MERGE_CONFLICT yet leave the tree untouched (auto-aborted); verify with `git status` + comparing HEAD vs origin/main SHAs before reacting. A non-force push that succeeds proves no remote commits were lost.
